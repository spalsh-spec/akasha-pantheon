Preliminary Design Review

Architecture · interfaces · requirements · test matrix · risks · sign-off.

Purpose

This PDR establishes that the design is sound enough to invest further engineering hours in. It documents what exists today (v0.2.0), the specific test verdicts that closed each requirement, and the residual risks that remain. It is the document an external reviewer signs after a 30-minute walkthrough.

1 · Scope

Akasha Pantheon is a single-host web application for asking philosophy / physics / cosmology questions through a structured-output interface to a large language model. Out of scope: multi-user authentication, RBAC, billing, distributed deployment, mobile native apps.

3.2 Data flow

  1. User types question → submits via click or ⌘+Enter.
  2. Frontend POSTs { question, provider } to /api/oracle.
  3. Server picks provider (FAST=Ollama, DEEP=Cloudflare), prepends system prompt, sends to backend.
  4. Server receives raw text response, returns JSON { raw, provider, model, ms, exported }.
  5. Frontend parses raw via the tolerant 6-key regex, renders 5 panels + meta line.
  6. Server writes the same raw as a markdown file to the iCloud directory (timestamped, slugged).

4.2 Storage interface

iCloud archive is opt-in via ICLOUD_EXPORT_DIR. Files are YYYY-MM-DDTHH-MM-SS-slug.md. Body is the question + metadata + raw response. No structured DB — flat markdown is the lingua franca of personal knowledge tools (Obsidian, Bear, Notion-import).

6 · Performance

FAST mode latency on M1 16GB (no other heavy apps): mean 14.8s, median 13.8s, max 20.1s. DEEP mode latency to Cloudflare: mean 5.6s, median 5.3s, max 6.2s. Both well under 30s reflective-use threshold.

Memory footprint: gemma3:4b loaded ~3.3 GB RAM resident; node server ~80 MB; browser tab ~200 MB. Total budget for the application stack ≤ 4 GB on a 16 GB machine — comfortable headroom for Slack, VS Code, Chrome simultaneously.

7 · Security

  • Default mode is air-gapped. FAST runs entirely locally; no network calls. Question text never leaves the machine.
  • DEEP mode sends question text to Cloudflare. Per Cloudflare's docs, request bodies are not retained for training. No PII is sent automatically; user-provided question content is the only payload.
  • API keys are stored in .env (gitignored). Never logged, never sent to the frontend.
  • Input validation: question is truncated to 1000 chars; JSON body capped at 64 KB.
  • Output rendering: all model outputs HTML-escaped before insertion into the DOM. No innerHTML with raw strings.

9 · Open questions for next review

  1. Should the format prior become user-editable (e.g. add a 6th section)? — Defer to v0.3.
  2. Should there be a "compare" view that runs FAST and DEEP simultaneously and renders side-by-side answers? — Useful for calibration research.
  3. Should answer corpus be searchable / RAG-able? — iCloud archive is already in markdown; obvious target.
  4. Should N be increased to 100 with cross-question variance reported? — Yes, for v1.0 paper.
